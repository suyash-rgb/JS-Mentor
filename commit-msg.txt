Phase 1: Data Ingestion for Analytics (Student Side)

This commit completes the first phase of the Analytics API implementation, focusing on the ingestion of real-time student activity data. This data will serve as the foundation for the upcoming Trainer Dashboard aggregations and the Machine Learning risk prediction model.

Summary of Changes:

1. Database Model Synchronization
Updated the ExerciseEvaluation model in app/models/learning.py to match the restructured database schema. Added fields for grading support including status (NEW, PENDING_REVIEW, GRADED), grade, feedback, graded_by, and graded_at.

2. Analytics Schemas
Created app/schemas/analytics.py defining strict Pydantic models for incoming ingestion payloads. This includes ProgressUpdate for tracking time spent on topics, ExerciseSubmission for logging code execution results, and QuizSubmission for recording performance scores.

3. Analytics Router and Endpoints
Implemented app/routers/analytics.py with three new protected POST endpoints. The progress endpoint handles student learning path tracking, the exercise endpoint records detailed code results, and the quiz endpoint logs performance metrics with automated pass/fail calculation. A require_student dependency ensures only authenticated students can report activity.

4. Application Integration
Registered the new analytics router in app/main.py, enabling the /api/v1/analytics route prefix for all ingestion operations.

This implementation allows the frontend to begin transmitting high-fidelity activity data, transitioning from a static learning experience to a metrics-driven platform.
